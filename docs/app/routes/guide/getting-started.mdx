# Getting Started

Install the package, add one config file, and run ESLint from the same directory as your `tsconfig.json`.

## Prerequisites

- Node.js >= 22
- ESLint >= 10
- TypeScript >= 5
- An existing `tsconfig.json`

<Note>
This config uses type-aware TypeScript rules by default. If ESLint cannot find your TypeScript project, run it from the package or workspace directory that owns `tsconfig.json`.
</Note>

## Create the ESLint config

Use `eslint.config.ts` when your toolchain can load TypeScript config files:

```typescript
// eslint.config.ts
import { getEslintConfig } from "eslint-config-setup"

export default await getEslintConfig({ react: true })
```

If you prefer a JavaScript config, use the same import from `eslint.config.js`:

```javascript
// eslint.config.js
import { getEslintConfig } from "eslint-config-setup"

export default await getEslintConfig({ react: true })
```

<Note>
Use `eslint.config.mjs` instead if your package is not ESM and does not set `"type": "module"` in `package.json`.
</Note>

## Add a lint script

Start with the smallest possible script:

```jsonc
// package.json
{
  "scripts": {
    "lint": "eslint ."
  }
}
```

Then run it:

## Recommended starter configs

### React app

```typescript
// eslint.config.ts
import { getEslintConfig } from "eslint-config-setup"

export default await getEslintConfig({ react: true })
```

React mode covers `react/*`, `react-hooks/*`, and `jsx-a11y/*`. When `ai: true` is also enabled, AI-only React performance checks run under `react-perf/*`. See [React Linting](/guide/react) for the rule surface.

### AI-assisted repo

```typescript
// eslint.config.ts
import { getEslintConfig } from "eslint-config-setup"

export default await getEslintConfig({ react: true, ai: true })
```

Use `ai: true` when generated code is expected to meet stricter conventions around explicit types, naming, magic values, and complexity. See [AI Mode](/guide/ai-mode) for details.

### Node package

```typescript
// eslint.config.ts
import { getEslintConfig } from "eslint-config-setup"

export default await getEslintConfig({ node: true })
```

Use `node: true` for backend packages, CLIs, scripts, workers, and libraries that run in Node.js.

### React + OxLint

```typescript
// eslint.config.ts
export default await getEslintConfig({ react: true, oxlint: true })

// oxlint.config.ts
import { getOxlintConfig } from "eslint-config-setup"
export default getOxlintConfig({ react: true })
```

```jsonc
// package.json
{
  "scripts": {
    "lint": "oxlint && eslint ."
  }
}
```

Use `oxlint: true` only when OxLint runs before ESLint. `getOxlintConfig()` keeps the OxLint side aligned with the ESLint options. See [OxLint Integration](/guide/oxlint) for the full split-linting setup.

## Configuration flags

Enable only the flags that match the project:

```typescript
export default await getEslintConfig({
  react: true,   // React 19+, Hooks, JSX-A11y, Storybook, Testing Library
  node: true,    // Node.js globals and rules
  ai: true,      // Strict rules for AI-assisted development
  oxlint: true,  // Disable rules covered by OxLint
})
```

Flags are independent — combine them however you need. See [Configuration](/guide/configuration) for details.

## Troubleshooting

### 1. Unsupported runtime or peer dependency

Check the versions ESLint actually sees:

```bash
node --version
npx eslint --version
npx tsc --version
```

Use Node.js >= 22, ESLint >= 10, and TypeScript >= 5.

### 2. Missing or misdetected `tsconfig.json`

Type-aware linting needs a TypeScript project. Make sure `tsconfig.json` exists and run the lint command from the directory that owns it. In a monorepo, prefer a package-local `eslint.config.*` next to the package `tsconfig.json`.

### 3. Editor and CLI disagree

If the CLI passes but the editor reports different diagnostics, check the editor working directory. Point the ESLint extension at the workspace or package root that contains `eslint.config.*` and `tsconfig.json`.

### 4. Browser and backend code share one repo

Do not rely on one global runtime when browser and Node.js code live together. Use directory-local configs or scoped config blocks so frontend files use `react: true` and backend files use `node: true`. See [Configuration](/guide/configuration#mixed-frontendbackend-repositories).

### 5. OxLint setup reports duplicate or missing rules

Run OxLint first, then ESLint:

```jsonc
{
  "scripts": {
    "lint": "oxlint && eslint ."
  }
}
```

Use `getOxlintConfig()` for `oxlint.config.ts` and enable `oxlint: true` in `eslint.config.ts`. If ESLint runs before OxLint, the split-linting setup cannot provide the intended feedback order.

## Customizing rules

Every config is fully customizable:

```typescript
import { getEslintConfig, disableRule, addRule } from "eslint-config-setup"

const config = await getEslintConfig({ react: true, ai: true })

disableRule(config, "@typescript-eslint/no-magic-numbers", { scope: "tests" })
addRule(config, "no-console", "off", { scope: "scripts" })

export default config
```

See the [Rule API](/guide/rule-api) for the full set of manipulation functions.
